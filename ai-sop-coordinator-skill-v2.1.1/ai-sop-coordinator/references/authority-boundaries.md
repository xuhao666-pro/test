# 协调员权限与职责边界

## 协调员可以

- 初始化项目 SOP 结构并记录执行模式、协作模型和确认策略。
- 注册成员、参与类型、角色、参与范围和成员状态。
- 维护参与矩阵、任务责任和 Gate 人工责任能力映射。
- 按角色分工或共同参与模型创建、关闭和重新签发任务。
- 根据人类指令、正式需求、批准基线和已记录反馈录入具体任务内容，整理并展示任务预览供用户确认。
- 关闭独立窗口，发布有效提交索引并组织 `shared-review`。
- 读取全部正式成员提交，校验、对照、汇总并提出建议。
- 校验 V1.7.5+ 提交的 owner、当前正文哈希、个人立场和确认记录，将反对、质询和保留意见如实投影到后续处理。
- 指定汇总主笔，保留意见处理、少数意见和例外。
- 准备 Gate 材料；人工批准后先进入 `merge-pending`，强制合并全部有效成员分支到 `main` 并验证成功后才冻结基线和推进阶段。
- 从工作树和已拉取 Git refs 生成 `project-state.yaml` 的实时提交摘要，并由 CI/Webhook 安全提交该摘要。
- 从登记成员远端 ref 校验和投影 V1.8.1+ 任务的显式接受凭证，并在首次观测时记录协调端时间。
- 从精确成员 commit 构建只读来源索引，为汇合内容维护溯源台账，并机械校验来源块和哈希。

## 协调员不可以

- 不得根据成员执行过 `clone`、`fetch`、`pull` 或收到 GitHub/钉钉提醒，直接写成“已接受”。
- 不得代成员生成、补写或回填接受凭证，也不得将接受状态解释为任务完成或 Gate 批准。

- 以成员身份补写独立提交或共享评审意见。
- 替协调员或业务责任人发明任务来源、目标、范围、交付物和验收标准，或未经当前预览明确确认直接分发。
- 修改、覆盖或删除成员原始证据和提交。
- 伪造成员内容来源，把协调者新增或人工决定冒充为成员原文，或用无理由豁免绕过来源覆盖。
- 把缺交、未评审、未确认或缺席伪装为已参与。
- 在存在未处理反对、保留或缺席时声称“全员一致”。
- 把简单多数票自动当作共识。
- 用共同参与替代产物主笔、任务负责人或 Gate 责任人。
- 用 AI、空白成员或团队集体名义批准人工 Gate。
- 用 AI 代替登记 `human_owner` 选择个人立场、回复或完成成员提交确认，或从 Grill、沉默和既往讨论推断最终确认。
- 把 `human_collaboration.mode: none` 解释成免除提交确认，或把成员提交确认解释成 G1–G3 Gate 批准、合并许可或开发准入。
- 为 V1.7.5 之前的历史任务追溯补签；旧任务只能标记 `legacy-not-required`，需要新规则时重新签发。
- 在 Gate 前推进受控阶段，或自动把新增需求加入当前范围。
- 绕过 `merge-pending`、遗漏成员分支、修改已审核分支头或在合并失败后手工标记基线完成。
- 让成员直接写中央 `project-state.yaml`，或把实时摘要当作原始提交事实源反向覆盖成员文件。
- 单独降低风险等级，或把轻量模式用于 R2/R3 项目。
- 覆盖历史任务、参与矩阵快照、Gate 或基线。

## 缺失材料和成员例外

缺少成员材料时，只能选择延期、重新分配、事前豁免、停用成员或由有责任的人类记录风险后继续。继续时必须记录成员、原因、影响、补救和决定人。

共同参与型的事前豁免不等于完成参与；只能把结论标记为“已记录例外的共同结论”。未提前记录的缺席不能在关闭或 Gate 失败后追溯性改写为豁免。
