# 协调员工作流

## 1. 初始化

1. 创建项目 SOP 目录、A00 状态和决策记录。
2. 分别选择 `execution_mode` 与 `collaboration_model`：
   - `execution_mode`: `standard` / `lightweight`
   - `collaboration_model`: `role-based` / `collective-participation`
3. 选择 `gate_confirmation_policy`：
   - `accountable-members`：全部已映射 Gate 责任人确认。
   - `all-participants`：全部有效人类参与成员确认，同时满足责任能力覆盖。
4. 登记成员卡、成员状态、参与范围和唯一 `git_branch`。默认成员分支为 `sop/member/<member-id>`，不得与 `main` 或其他成员分支重复。
5. 分别登记 G1–G3 的业务、产品、项目、技术、测试和专项风险人工责任能力。
6. 记录最高风险等级、真实开发状态、下一 Gate 和阻塞项。R2/R3 自动使用标准模式，并登记专项人类责任人。
7. 初始化 `submission_tracking` 实时投影；分别统计成员提交状态和提交前 human owner 确认状态，所有协调者状态操作结束后自动刷新。
8. 初始化阶段来源索引、内容溯源台账和报告位置；新项目启用强制溯源且禁止无来源历史标签。

`execution_mode` 决定流程严谨度和产物合并程度，`collaboration_model` 决定谁参与、如何分发和如何评审。不得从一个字段推断另一个字段。

初始化脚本只检测 Git，不执行 `git init`。先建立团队共享 Git 仓库并创建 `main` 与各成员登记分支。成员自行 commit/push；Gate 人工通过后由协调员命令强制 commit Gate 状态并合并分支。

## 2. 分离三类责任

每轮开始前分别确定：

- 参与：应完成独立提交、公开评审和 Gate 确认的成员。
- 任务责任：每个产物的负责人、主笔、协作成员和审查人。
- Gate 责任：具备相应责任能力并承担人工决定的人类成员。

同一成员可以参与全部工作、负责多个任务或承担多项责任能力。不得用全员参与替代主笔和责任人，也不得用固定角色阻止成员提出跨专业意见。

## 3. 分发规则

新签发且精确绑定 Member V1.8.1+ 的任务必须携带 `acceptance_policy`。成员拉取任务后先运行 `accept-assignment`，将接受凭证提交并推送到登记分支。协调端只能从该精确远端 ref 投影 `pending`、`accepted` 或 `invalid`；单纯拉取仓库和通知送达不可审计，不构成接受。

每个任务先由协调员依据人类指令、需求池、批准基线、评审待办或已确认回流记录输入具体内容。至少输入任务来源、目标、范围内事项、交付物和验收标准；脚本只派生任务 ID、成员资料、Git 分支、基线、权限和标准质量检查，不得代替协调员创造任务。

需求分析任务需要登记真实成员进行动态访谈时，协调员在预览中显式选择 `human_collaboration.mode: adaptive-grill` 并确认问题上限。该模式只允许 `requirement-analysis + isolated-discovery`；任务哈希同时保护协同配置。未显式授权时，成员不得把普通讨论登记为正式人机协同证据。

`human_collaboration` 只控制 Grill。Coordinator V1.7.6 新任务无论使用 `none` 还是 `adaptive-grill`，都必须写入独立的 `submission_confirmation` 策略，要求登记 `human_owner` 在 submit 前确认当前 `main-output.md` 正文哈希和个人立场。该确认只允许成员贡献提交，`gate_effect` 固定为 `none`；AI 不得代选立场、代签或把 Grill 结果当作最终确认。

首次执行创建命令只生成预览和 `confirmation_token`，不写分发文件或阶段状态。协调员向用户展示完整预览；只有用户明确确认后，才使用相同参数和 `--confirm-dispatch <token>` 正式写入。任一字段变化都使旧 Token 失效并要求重新确认。

### 角色分工型

按主要角色、附加角色、专业能力和任务责任创建个人任务。只要求 A00 列出的应参与成员提交和评审。

### 共同参与型

1. 在参与矩阵登记当前全部有效成员为应参与成员，必要时在窗口打开前记录豁免。
2. 为每名应参与成员创建独立任务；同一任务目标仍产生独立提交，不产生共享可写原稿。
3. 在独立窗口关闭前隔离成员结论。
4. 关闭窗口后同时公开全部有效提交和索引。
5. 创建指向已关闭独立轮次的 `shared-review` 轮次，由全部应参与成员分别提交确认、反对、质询和保留意见。
6. 由明确主笔或协调员形成候选共识产物，不改写原始提交。

已分发任务不得原地修改；需要调整时签发新任务或新 `assignment_version`。

## 4. 阶段执行

### 阶段 A

角色分工型向指定需求分析成员分发 `requirement-analysis`；共同参与型向全部应参与成员批量分发。所有成员自行选择来源和方法，独立提交 A01。关闭窗口并完成公开评审后，形成 A02、A03、A04 和 G1 材料。

### 阶段 B

按两轮执行：

1. 在同一首轮并行分发 `function-design` 和 `system-inventory`。共同参与型要求全部应参与成员分别对两类任务作出提交；没有系统权限时提交无法访问说明和待核实项，不猜测事实。
2. 关闭首轮后公开 A05/A06，创建独立的多方案 `shared-review` 轮次；每份评审任务使用 `review_of_round` 指向被评审首轮。
3. 首轮评审完成后再分发 `prototype-validation`。输入仅限公开的共识设计草案、系统盘点和待验证问题。
4. 关闭验证轮后创建指向该轮的 `shared-review`，由全部应参与成员评审验证结果、范围影响和生产差距；关闭评审轮后再处理反馈回流，形成 A07 并准备 G2。

### 阶段 C

角色分工型按产品、技术、测试和项目责任分发；共同参与型向全部应参与成员收集产品、技术、测试、风险和任务拆分建议。由指定主笔形成 A08/A09 候选稿，再通过指向相应实质工作轮的 `shared-review` 由应参与成员交叉评审。两种模型都必须为候选产物和开发任务指定责任人；G3 前不开始生产实现。

## 5. 收集、参与矩阵与汇总

1. 拉取成员分支并校验身份、协议、schema、协作模型、基线、文件和状态。
2. 对 V1.7.5+ 任务校验 `human-submission-confirmation.yaml` 与当前正文哈希、登记 owner、个人立场和提交清单摘要一致；未确认或 stale 时不得标记 `valid`。旧任务只记录 `legacy-not-required`，不得追溯补签。
3. 维护 `aggregation/participation-matrix.yaml`，逐成员记录独立提交、公开评审、Gate 确认、任务责任、豁免/缺席原因和影响。
4. 单独记录缺交、迟交、无效提交和带风险继续，不代写。
5. 运行 `build-source-index`，从每个成员分支的精确 commit 验证并汇总内容块索引。
6. 先建立证据、需求追踪、成员贡献来源和意见处理矩阵，再形成结论。
7. 保留事实冲突、理解差异、方案差异、新增范围、少数意见和未决风险。
8. 为汇总产物指定主笔，并声明可用的共识等级。
9. 在汇合稿逐块加入 `P` 标识，用 `trace-content` 登记形成方式和全部成员源块，完成共同评审后运行 `validate-provenance`。
10. 每次任务、校验、关闭、评审、Gate 或回流状态变化后，把预期、已提交、有效、待校验、无效、缺失和提交确认状态重新投影到 `project-state.yaml`。

共同参与型只有全部要求完成时才可声明“全员一致”。存在有效事前豁免时，只能声明“已记录例外的共同结论”；存在未处理缺席、反对或保留意见时不得声称全员通过。

## 6. 实时提交投影

刷新状态时同时读取任务对应的接受凭证。首次观测到有效凭证时记录协调端 `observed_at`；成员填写的 `accepted_at` 只作为成员声明时间保留。接受状态只表达成员已接单，不改变提交校验、收轮、Gate、合并或基线状态。

- 工作区内提交由所有协调者命令自动刷新。
- 跨环境成员 push 后，合并式 GitHub Actions 工作流读取远程成员 ref；无需把成员分支提前合并到 `main`。
- CI 在干净 `main` 上运行 `refresh-project-state --fetch --validate-remote`，并在隔离 worktree 中用受信任 Member CLI 校验提交。
- 远程文件完整但尚未完成校验时标记为 `pending-validation`；隔离校验通过后标记为 `valid`，失败时记录明确原因。
- 每个任务投影提交确认是否 required、状态、登记 owner、个人立场、确认时间和正文哈希；阶段及全项目分别汇总 required、confirmed、pending、attention 和 legacy 数量。
- `oppose`、`question` 或 `reserve` 可以通过提交校验，但必须标记为需要后续处理，不得显示为共识或 Gate 通过。
- 投影内容没有语义变化时保持原 revision 和更新时间，定时兜底不得制造空提交。
- `project-state.yaml` 是摘要投影，原始任务、提交清单、输出文件、阶段状态和参与矩阵仍是事实源。
- 只有协调员或 CI 写中央状态；成员只维护自己分支中的提交清单，避免状态文件冲突。

## 7. 老板状态看板

- 私有仓库使用仓库根 `README.md` 嵌入 `dashboard/status.svg` 作为老板看板，不部署 Pages、不读取公共 Raw 地址。
- 运行 `install-dashboard` 后立即生成首版 README 看板，同时安装合并式工作流、Coordinator CLI、Member CLI 和标准库渲染脚本；提交后老板打开仓库首页即可查看。
- 登记成员分支 push、中央状态变化、手工触发或定时兜底均进入同一工作流；工作流在一个 commit 中更新中央状态、README 和 SVG，不串联依赖第二个 push 工作流。
- 工作流使用仓库内置 Token，只申请提交看板所需的 `contents: write`；不得把个人 Token 写入仓库或浏览器。
- 若分支保护禁止机器人直推，协调员应在中央状态提交前运行仓库内渲染脚本，把 README 和 SVG 放入同一次受审提交，或采用团队批准的 PR 流程。
- 看板是只读摘要，不作为任务、提交、Gate 或基线的事实源，也不得从 README 或 SVG 反向修改中央状态。

## 8. 文档内容来源追踪

1. 成员 `submit` 自动生成并冻结 `content-block-index.yaml`。
2. 协调员运行 `build-source-index`，直接读取成员 Git ref，无需预先合并分支。
3. 主笔形成汇合稿时，为每个实质内容块添加唯一 `P-001`；综合多个成员时引用全部源块。
4. 运行 `trace-content` 写入台账；原文、改写、综合、推导、冲突保留、人工决定和协调者新增必须如实分类。
5. 共同评审确认来源选择和形成方式，台账状态至少为 `reviewed`。
6. 运行 `validate-provenance` 和 `provenance-report`。内容无标识、标识无台账、源块不存在、commit/哈希变化、元数据伪造或未审核均阻塞 Gate。

历史项目迁移后可对无法还原的既有基线使用 `legacy-unattributed`，但必须填写原因并完成人工审核；新内容不得使用该标签。

## 9. shared-review 基线与错误轮次

1. 创建 shared-review 前，被评审轮次必须已关闭且 `submission-index.yaml` 必须存在、可解析并匹配阶段和轮次。
2. 阶段 A 默认只发布 submission index；不存在的 summary 不得进入任务。
3. summary 只有非空且不含 `[[FILL]]` 时才可进入任务；需要 summary 的阶段 B 后续任务在其未完成时必须失败。
4. 成员端在 inspect/init/validate/submit 中统一验证全部 baseline refs，不允许成员创建中央缺失文件或忽略错误。
5. 已分发错误任务不得原地编辑。运行 `supersede-round --replacement-round ...` 保留旧任务和决定记录，再创建新轮次。

## 10. Gate

1. 在 G1、G2、G3 或风险触发点暂停受控工作。
2. AI 生成审查包和建议，不填写人工批准结论。
3. 区分成员提交确认与 Gate 确认：前者只确认一份成员正文及个人立场，不能满足后者的责任能力、参与策略或审批结论。
4. 强制校验文档来源覆盖率 100%、源块/commit/哈希一致并已评审。
5. 校验确认策略、该 Gate 必需责任能力和真实人类身份。
6. `all-participants` 下校验全部有效人类参与成员均明确确认；反对、保留或缺失不能记为全员通过。
7. R2/R3 同时校验专项人类责任人；共同参与或全员确认不能替代专项责任。
8. `prepare-gate` 冻结全部有效成员分支的 `expected_head`，人工审核必须覆盖这些 commit。
9. 人工通过后运行 `approve-gate`，只把阶段置为 `merge-pending`，不冻结基线、不推进阶段。
10. 在干净且已检出的 `main` 上运行 `merge-approved-gate`。命令先在临时集成分支合并全部成员分支；任一冲突时保持 `main` 不变。
11. 全部分支合并成功并验证每个冻结 commit 都是 `main` 的祖先后，才记录证据、冻结基线并推进阶段。退回时保留历史并创建新任务版本。

## 11. Git 原则

- `main` 保存分发、A00、参与矩阵、汇总状态、成员分支合并结果和批准基线。
- 成员只提交自己的分支和目录。
- 独立窗口关闭前不把成员结论合并到公开材料。
- `prepare-gate` 记录全部有效成员分支和精确 commit；审核后分支头变化将使批准失败。
- 人工批准只进入 `merge-pending`；不得用普通状态转换、手工复制文件或只合并协调员汇总分支来代替全部成员分支合并。
- 强制集成使用临时分支验证，全部成功后快进 `main`；冲突、缺分支或祖先校验失败均阻塞基线。
- G3 后的代码任务使用独立功能分支或 worktree。
