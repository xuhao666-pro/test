# Gate 规则

## 责任能力

固定 Gate 校验责任能力，不用日常参与角色替代人工决策责任：

| Gate | 决策 | 最低人工责任能力 |
| --- | --- | --- |
| G1 | 需求合同是否足以进入功能设计 | `business-decision`、`product-decision` |
| G2 | 方案和 P0 范围是否可以冻结 | `business-decision`、`product-decision`、`technical-decision` |
| G3 | 是否允许进入正式开发 | `project-decision`、`product-decision`、`technical-decision`、`test-decision` |

同一真实人类可以承担多项责任能力。AI 身份、AI 角色、空白成员或已停用成员不能成为人工批准人。R2/R3 还必须覆盖 A00 登记的安全、数据、合规或其他专项责任能力。

## 确认策略

### `accountable-members`

要求映射到当前 Gate 的全部责任人类成员确认。适合角色分工型，也是旧项目迁移的兼容默认值。

### `all-participants`

要求当前全部有效人类参与成员确认，同时校验最低责任能力和专项责任能力已映射到具体人类。适合共同参与型；全员签字不能替代技术、安全或测试责任。

不得在 Gate 失败后追溯性排除反对或未确认成员。改变策略或成员范围须在审查前写入 A00 决策日志并更新参与矩阵。

## 成员提交确认不是 Gate

`human-submission-confirmation.yaml` 只表示登记 `human_owner` 确认某个 `main-output.md` 正文哈希准确记录本人立场，并同意把它作为个人贡献提交。其 `gate_effect` 必须为 `none`。

它不能替代本文件定义的 Gate 责任能力、确认策略、逐人审批、分支头审核、合并许可或基线冻结。即使所有成员提交均已确认，也必须独立完成 G1–G3 审核；个人立场为 `oppose`、`question` 或 `reserve` 时尤其不得据此声称全员通过。AI 不得代签两类人工确认。

## 参与和共识约束

- `all-participants` 下任一应确认成员缺失、反对或保留时，不得记录为全员通过。
- 共同参与型缺少独立提交或公开评审时，Gate 包须明确例外和影响。
- 事前有效豁免允许形成“已记录例外的共同结论”，但不得标记为“全员一致”。
- `accountable-members` 通过不表示全员一致；只表示责任成员完成批准。
- P0 追踪完整不能弥补参与矩阵缺失。
- 原始 `SRC` 追踪完整不能替代成员贡献来源追踪；后者同样要求 100% 实质内容覆盖。

## 内容来源门禁

`prepare-gate` 前必须满足：

- 每个成员提交都有与正文一致的内容块索引。
- 汇合产物的每个实质内容单元都有唯一 `P` 标识或带理由的结构豁免。
- 每个 `P` 标识都有台账记录，且指向当前来源索引中的真实源块，或指向明确人工决定；实质内容不得用豁免绕过。
- 台账保存的成员、任务、源文档、分支、commit、内容哈希和 `SRC` 与来源索引一致。
- `synthesis` 保留全部实际使用的成员来源；冲突保留项不因未被采用而删除。
- 全部台账记录完成共同评审；覆盖率为 100%，无孤立标识、孤立台账或失效哈希。

任一来源分支头、正文或块哈希变化都会使现有台账失效。不得仅修改台账中的 commit 或哈希来“修复”校验，必须重新构建来源索引、重新确认汇合内容并再次评审。

## Gate 结论

- `pass`
- `conditional-pass`
- `return`
- `terminate`

只有 `pass` 和有效 `conditional-pass` 可以进入 `merge-pending`。每个条件必须包含 `owner`、`due_at` 和 `overdue_action`。影响产品范围、数据安全、技术可行性或开发安全的阻塞项不得条件通过。即使人工结论通过，也必须完成强制分支合并后才能冻结基线。

## 人工通过后的强制合并

- Gate 准备时记录全部有效成员的 `member_id`、登记分支和 `expected_head`。
- 人工批准必须显式设置 `human_review_covers_branch_heads: true`，表示审核覆盖所列精确 commit。
- `approve-gate` 只记录人工通过并进入 `merge-pending`。
- `merge-approved-gate` 必须把全部冻结成员分支合并到配置的 `main`，验证每个 `expected_head` 都是合并后 `main` 的祖先，并记录合并前后 commit。
- 分支缺失、分支头变化、重复分支、合并冲突、非干净工作区、非 `main` 工作区或祖先校验失败时不得冻结基线。
- 不允许通过删除成员、改写 `merge_plan`、手工复制产物或普通状态转换绕过合并；如审核对象变化，退回并重新准备 Gate。

## 风险触发 Gate

数据库迁移、公共 API、权限与租户、支付订单、隐私和上传、生产配置和密钥、真实数据验证，以及 G2 后 P0 或核心流程变化，触发专项人工审查。

专项 Gate 只冻结受影响工作，结论进入决策记录并在下一固定 Gate 引用。AI 可以建议提高风险等级，不得单独降低。共同参与、全员确认或多人共同签字不能稀释 R2/R3 专项责任。

## Gate 材料

材料必须列出：

- 基线版本、执行模式、协作模型和确认策略。
- 应参与成员、实际参与状态、缺席/豁免原因和影响。
- 共识声明等级、反对和保留意见。
- 最低责任能力、映射成员和实际人工确认。
- 每名人工批准人的 `member_id`、`name`、`role` 和 `approved_at`；共同参与型无固定角色时 `role` 可为 `general-contributor`。
- 最高风险、专项责任人、已确认项、非阻塞项和阻塞项。
- 建议结论、条件、回退位置和下一步。
- 目标 `main`、全部成员分支、冻结 commit、人工是否覆盖这些 commit，以及合并状态。
- 来源索引哈希、内容溯源台账、覆盖报告、无来源/冲突/豁免项及其审核结论。
