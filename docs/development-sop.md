# 开发交付阶段 SOP

> 适用起点：G3 开发准入基线完成冻结后
> 适用终点：生产发布完成、观察期结束并关闭交付
> 文档状态：团队执行基线建议稿

## 1. 文档目的

本 SOP 适用于项目通过开发准入 Gate 后，从正式编码开始，到代码评审、集成、发布、监控、回滚和复盘结束的全过程。

本 SOP 不替代需求分析、方案验证和开发准入流程。原型、演示代码和探索性测试只有在开发准入 Gate 明确批准后，才能转化为生产实现输入。

本项目中的 G3 等价于“允许按已批准任务包进入真实代码开发”，并不等于代码已经完成，也不等于允许直接发布。G3 后仍须依次完成实现、代码审查、集成验证、发布准入、灰度和发布后验证。

## 2. 适用范围

适用于以下工作：

- 普通业务功能开发。
- 前端、后端、小程序、管理端和公共组件修改。
- 数据库、公共 API、权限、隐私、订单、支付、上传、定时任务等高风险修改。
- 缺陷修复、性能优化和兼容性改造。
- 灰度发布、正式发布、回滚和事故处置。

不适用于以下工作：

- 尚未通过开发准入 Gate 的需求探索和原型验证。
- 未经授权的生产操作、数据删除、密钥修改或生产数据库变更。
- 与已批准任务包无关的重构和范围扩张。

## 3. 核心原则

1. **先准入，后开发**：未通过开发准入 Gate，不得修改生产目标代码。
2. **一项任务一名主责**：共同参与不等于多人重复实现同一功能。
3. **范围最小化**：只修改任务包允许的模块和文件，不夹带无关重构。
4. **测试先行**：行为变化前先形成测试用例或测试矩阵。
5. **证据优先**：代码、测试、CI 和运行结果是完成证据，口头说明不能替代。
6. **高风险升级**：权限、数据库、隐私、订单、支付、生产配置和密钥必须经过专项人工审查。
7. **可停止、可回滚**：发布前必须明确停止条件和可执行回滚路径。
8. **人工承担生产责任**：AI 可以计划、实现、检查和建议，但不能单独批准生产行为。
9. **机器文件退出主视图**：状态、哈希、索引和日志由工具维护，人工重点审查代码、测试和决策。
10. **无进展必须熔断**：没有新的前提、方法或责任变化，不得重复签发相同任务。

## 4. 角色与责任

| 角色 | 主要责任 | 不得替代的责任 |
| --- | --- | --- |
| 产品/业务责任人 | 确认范围、验收标准和业务取舍 | 技术实现和安全批准 |
| 技术负责人 | 审核架构、接口、数据、依赖和技术风险 | 产品范围决定 |
| 开发主责 | 编写计划、测试和代码，完成自检并提交 | 自行批准高风险变更 |
| 代码审查人 | 审查正确性、安全性、兼容性、测试和范围 | 代替开发者补写提交 |
| 测试责任人 | 维护测试矩阵、系统测试和发布验证 | 产品或技术 Gate 批准 |
| 专项责任人 | 审核权限、食安、隐私、支付等专项风险 | 被多数意见覆盖 |
| 发布负责人 | 组织灰度、监控、停止和回滚 | 单独改变已批准发布范围 |
| 协调者 | 维护任务、状态、证据、Gate 和阻塞项 | 冒充成员或代签人工确认 |

一名真实成员可以承担多个角色，但每项具体责任必须映射到明确的人。

## 5. 总体流程

```text
G3 人工批准
  → 冻结成员提交合并与祖先校验
  → G3 基线冻结（development-entry-approved）
  → 任务拆分与确认
  → 成员接受
  → 实施计划
  → 测试先行
  → 正式实现
  → 本地与 CI 检查
  → 代码审查
  → 集成验证
  → G4 发布准入审查
  → 灰度发布
  → 监控与回滚
  → G5 交付关闭与复盘
```

建议的任务状态：

```text
draft
→ ready-for-confirmation
→ distributed
→ accepted
→ in-progress
→ submitted
→ under-review
→ changes-requested / approved
→ integrated
→ released
→ verified
→ closed
```

### 5.1 Gate 与日常任务的边界

| 控制点 | 审查对象 | 默认参与人 | 通过后的权限 |
| --- | --- | --- | --- |
| G3 开发准入 | 最终方案、测试矩阵、开发任务包、风险与回滚 | Gate 策略要求的参与者和责任人 | 可以执行批准范围内的真实开发 |
| 单任务代码审查 | PR、代码、测试和任务验收证据 | 开发主责之外的审查人；高风险任务增加专项责任人 | 可以合并该任务的精确提交 |
| G4 发布准入 | 已集成版本、系统测试、迁移、监控、灰度和回滚 | 项目、技术、测试、发布及相关专项责任人 | 可以按批准方案灰度发布 |
| G5 交付关闭 | 生产验证、观察期指标、事故和遗留项 | 项目、业务、技术、测试及发布责任人 | 可以关闭本次交付 |

常规开发任务不要求所有成员重复实现或审查。`all-participants` 适用于项目 Gate，不应机械扩展到每个 PR；单任务采用“一名主责 + 至少一名独立审查人 + 按风险追加专项责任人”。

## 6. D0：开发准入

### 6.1 进入条件

正式开发前必须确认：

- G3 人工结论为 `pass`，或当前任务明确位于 `conditional-pass` 允许范围内。
- G3 已完成 `merge-pending`：冻结的有效成员提交已合并，且精确提交是目标主干祖先。
- G3 基线已经冻结并具有不可歧义的版本或提交引用。
- 项目状态已更新为 `development-entry-approved`；仅有通知、拉取、提交或口头同意不能替代这一状态。
- 需求范围、非目标和 P0 验收标准已批准。
- 目标代码仓库、技术栈和负责模块已明确。
- 系统盘点、原型结论和生产化差距已记录。
- 正式技术方案、接口和数据边界已批准。
- 数据库迁移与回滚策略已批准；不涉及数据库时明确记录“不涉及”。
- 权限、隐私、订单、支付、上传、日志、监控等风险已经评审。
- 测试矩阵和开发任务包已经批准。
- 灰度方案、停止条件和回滚路径已有草案。
- 生产代码与原型代码的边界清楚。

### 6.2 Gate 结论

仅允许以下结论：

- `pass`：允许进入正式开发。
- `conditional-pass`：仅允许执行列明条件内的开发任务。
- `reject`：退回需求、设计、盘点或验证阶段。

未获得真实人工批准时，状态必须保持 `not-started`。

### 6.3 G3 向开发阶段的交接清单

协调者在签发首批开发任务前应固定以下输入：

- `baseline_ref`：G3 冻结基线的精确版本和提交。
- 需求、P0 验收标准、最终方案和测试矩阵引用。
- 已批准开发任务包及其依赖顺序。
- 原型结论和“原型到生产”的剩余差距；不得把原型提交直接视为生产代码。
- 项目、产品、技术、测试及专项责任人映射。
- 当前稳定成员 Skill 版本和任务绑定规则。
- 目标主干、分支命名、合并策略和必需 CI 检查。
- 发布负责人、监控指标、停止条件和回滚草案。

以上任一关键输入缺失时，应创建准备或修复任务，不得用一个宽泛的开发任务包吸收缺口。

## 7. D1：任务拆分与确认

每个任务包必须足够小，可以由一名主责成员完成，并由其他成员独立审查。

### 7.1 必填字段

```yaml
task_id: DEV-001
title: ""
assignment_version: "v1.0"
baseline_ref: "G3-V1.0@<commit>"
required_member_skill: ""
requirement_refs: []
acceptance_refs: []
blocker_refs: []
primary_owner: ""
reviewers: []
risk_owners: []
risk_level: "R0"
goal: ""
allowed_scope: []
forbidden_scope: []
input_refs: []
expected_outputs: []
acceptance_criteria: []
test_requirements: []
dependencies: []
prerequisite_status: ready
expected_blocker_delta: []
target_branch: "main"
working_branch: ""
stop_condition: ""
rollback_considerations: ""
```

### 7.2 拆分规则

- 一个任务只解决一个清晰目标或一组强耦合目标。
- 数据模型、接口、前端交互、后台能力和测试可独立时分别拆分。
- 公共 API、数据库、权限和发布配置变更必须显式列出。
- 不得使用“完成整个功能”“处理所有问题”等不可审查任务。
- `allowed_scope` 之外的修改必须停止并重新确认任务。
- 已分发任务不得原地扩大范围；需要调整时签发新版本或替代任务。
- 前置依赖未集成时，后继任务不得标记为 `ready`。
- 一个功能可以拆成多个任务，但每条 P0 验收标准必须能追踪到至少一个实现任务和一个测试任务或测试项。

### 7.3 共同参与规则

- 每项实现由一名主责成员完成。
- 其他成员从产品、业务、技术、测试或专项风险角度独立审查。
- 不要求所有人重复搭建环境或重复实现同一代码。
- 全员 Gate 确认不能替代具体技术或专项责任人的批准。
- 开发主责不得作为该任务唯一代码审查人。
- 涉及数据库、权限、隐私、食安、支付、订单或生产配置时，必须绑定相应专项责任人。

## 8. D2：成员接受与实施计划

成员开始工作前应：

1. 获取任务中指定的精确 `baseline_ref`，不得自行换成另一个本地提交。
2. 核对任务版本、成员 Skill 要求、允许范围、禁止范围和依赖。
3. 在登记的功能分支或独立 worktree 中工作，记录分支与基线提交。
4. 以项目规定的接受回执确认任务；仅执行 `fetch`、`pull` 或打开任务文件不等于已接受。
5. 检查开发环境和测试入口是否可用。
6. 明确将新增或修改的文件。
7. 明确核心逻辑、数据读写、错误处理和权限检查。
8. 明确需要先编写的测试。

看板可以分别展示 `已拉取` 和 `已接受`：前者是机器观测，后者必须来源于成员针对精确任务版本的有效接受回执。

如果环境、依赖、权限或输入尚未准备好，应把任务标记为 `blocked`，创建前提准备任务，而不是反复尝试相同实现。

## 9. D3：测试先行

行为变更前必须先建立测试用例或测试矩阵，至少覆盖：

- 正常成功路径。
- 输入无效和依赖失败路径。
- 未登录、无权限和跨对象访问。
- 空值、极值、重复提交和过期状态。
- 数据一致性和状态转换。
- 幂等、并发和重试风险（适用时）。
- 旧版本兼容和回归场景。
- 高风险模块的专项负向场景。

每个测试应关联需求或验收编号。禁止通过删除测试、弱化断言、跳过权限或吞掉异常使检查通过。

## 10. D4：正式实现

### 10.1 实现要求

- 采用符合现有架构的最小变更。
- 优先复用现有模式和公共能力。
- 不新增不必要依赖。
- 不硬编码密钥、账号、Token 或生产配置。
- 不读取或写入未经授权的真实用户数据。
- 不修改未列入任务包的公共接口、数据库或部署配置。
- 所有数据写入均考虑失败、重试、一致性和审计。
- 权限检查必须在可信服务端边界执行，不能只依赖前端隐藏。
- 原型代码不得未经评审直接作为生产实现。

### 10.2 AI 权限等级

| 等级 | 典型操作 | 要求 |
| --- | --- | --- |
| L0 | 解释代码、回答问题 | 无写入 |
| L1 | 生成文档、测试和计划 | 开发者核对 |
| L2 | 修改普通业务代码 | 开发者审查 |
| L3 | 修改数据库、权限、登录、隐私、订单或支付 | 技术负责人和专项责任人批准 |
| L4 | 部署生产、删除数据、修改密钥 | AI 不得独立执行，必须由授权人操作 |

## 11. D5：本地与 CI 质量检查

按项目技术栈执行适用检查：

- 格式化和 lint。
- 类型检查。
- 单元测试。
- 集成测试。
- 端到端测试。
- 构建。
- 依赖、安全和密钥扫描。
- 数据库迁移的升级、兼容和回滚演练。

检查结果必须记录：

- 精确命令。
- 环境与版本。
- 通过、失败和未运行项。
- 失败原因及影响。
- 对应任务版本、提交 SHA 和 CI 运行链接。

局部检查通过不能替代任务要求的完整检查。无法运行的项目应标记为 `not-run`，不得描述为通过。

## 12. D6：提交与代码审查

### 12.1 开发提交内容

人工重点提交和阅读：

- 代码差异。
- 新增或修改的测试。
- 简短实施说明。
- 检查命令和结果。
- 已知风险和回滚提示。
- 精确提交 SHA、目标分支以及关联任务、需求和验收编号。

状态、哈希、索引、CI 日志和任务投影由工具维护，不要求人工逐份审读。

### 12.2 审查等级

- `P0`：必须修复，禁止合并。
- `P1`：原则上合并前修复；延期必须有责任人和期限。
- `P2`：可选优化，不阻塞当前交付。

### 12.3 审查清单

审查人至少核对：

- 是否满足全部关联验收标准。
- 是否超出任务允许范围。
- 异常流程和边界条件是否完整。
- 认证、授权和数据可见性是否正确。
- 数据迁移、并发、幂等和兼容性是否安全。
- 日志、监控和错误信息是否合适且不泄露敏感数据。
- 是否存在不必要复杂度、重复逻辑或性能问题。
- 测试是否真正覆盖修改行为。
- 回滚是否可执行。

高风险任务不能仅由开发者自审后合并。

### 12.4 审查与返工规则

- 审查结论必须绑定任务版本和精确提交；提交发生变化后，受影响结论必须重新验证。
- `changes-requested` 应列出问题等级、证据、责任人和再次提交条件。
- 修复应通过新提交完成，不改写已经归档的审查事实。
- 审查人确认的是代码与证据，协调者只负责汇总和状态投影，不能代签。
- 同一问题连续两轮无实质进展时，按第 18 节熔断并重新规划。

## 13. D7：集成验证

代码批准后进入集成流程：

1. 确认分支基于最新主干或可安全集成。
2. 解决冲突并重新运行受影响检查。
3. 合并已批准的精确提交，不以复制文件代替正式合并。
4. 在主干或候选发布分支运行集成测试。
5. 验证核心流程、权限、数据和兼容性。
6. 对失败项创建明确修复任务，不直接改写历史审查结论。
7. 更新主干上的任务状态、提交可达性和需求追踪投影。

集成完成条件：

- 所有 P0 问题关闭。
- 必需 CI 检查通过。
- 任务提交是主干祖先。
- 需求到代码、测试和审查结论可追踪。
- 未决 P1/P2 已登记责任人、影响和计划。

协调者只能将“已批准且已验证可达”的提交计入集成完成数。远端分支存在、成员声称完成或 PR 已创建，都不能替代主干祖先校验。

## 14. D8：G4 发布准入 Gate

发布前必须确认：

- 代码已集成且 CI 通过。
- 数据库迁移和配置变更已复核。
- Feature Flag 或灰度策略可用。
- 技术、业务和用户体验指标已配置。
- 告警阈值和通知责任人已配置。
- 停止条件明确。
- 回滚步骤已演练或确认可执行。
- 客服、运营和相关责任人已知晓影响。
- 发布负责人和观察窗口已确定。

发布 Gate 未通过时，不得通过手工绕过 CI 或临时修改配置直接上线。

G4 仅允许以下结论：

- `pass`：允许按已批准的灰度计划发布。
- `conditional-pass`：仅允许在明确条件、范围、期限和责任人约束下发布；权限、隐私、食安、数据一致性及不可回滚问题不得带条件放行。
- `reject`：退回开发、测试、集成或发布准备阶段。

G4 审核材料应以一个人类可读发布包为主，至少说明版本、改动范围、P0 验收结果、未决问题、迁移、监控、灰度、停止和回滚。CI 日志、哈希和索引作为机器证据链接，不要求成员逐份审查。

## 15. D9：灰度发布、监控与回滚

### 15.1 建议灰度顺序

```text
内部用户
→ 1%
→ 5%
→ 20%
→ 50%
→ 100%
```

团队可以按风险调整比例，但每次扩量前必须完成上一阶段观察。

### 15.2 监控指标

技术指标：

- 请求量、错误率和延迟。
- 崩溃率、CPU、内存和数据库负载。
- 慢查询、队列积压和异常日志。

业务指标：

- 核心流程完成率。
- 登录、订单、支付等适用流程成功率。
- 转化、退出和投诉率。

用户体验指标：

- 页面加载时间。
- 操作失败数。
- 用户反馈和客服问题。

### 15.3 停止条件

出现以下任一情况应暂停扩量并评估回滚：

- 错误率或核心接口延迟显著上升。
- 核心业务流程异常。
- 数据写入、状态转换或一致性异常。
- 权限绕过、隐私泄漏或食安风险。
- 投诉集中出现或异常日志显著增加。
- 无法解释的监控指标变化。

### 15.4 回滚顺序

优先采用低风险方式：

1. 关闭 Feature Flag。
2. 回滚前端资源。
3. 回滚后端服务。
4. 回滚数据库变更。
5. 执行经过批准的数据修复。

数据库回滚和数据修复属于高风险操作，必须由授权人批准并执行。

## 16. D10：G5 交付关闭

至少执行以下观察：

- 发布后 15 分钟：明显错误和崩溃。
- 发布后 1 小时：核心技术与业务指标。
- 发布当天：用户反馈、异常日志和告警。
- 后续 3 天：长期指标和边缘问题。

交付关闭条件：

- 核心冒烟测试通过。
- 技术、业务和体验指标处于可接受范围。
- 未发生未处理的 P0/P1 事故。
- 文档、API、数据库和运维说明已更新。
- 遗留问题已进入任务池。
- 发布结论由真实责任人确认。

满足关闭条件后形成 G5 交付结论并冻结发布基线。若观察期内存在未关闭 P0/P1 事故、数据修复未完成或风险责任人保留意见，状态保持 `released` 或 `verifying`，不得标记为 `closed`。

## 17. 事故处理与复盘

发生生产事故时：

1. 判断影响范围。
2. 暂停发布或扩量。
3. 必要时关闭功能或回滚。
4. 保存日志和证据，避免泄露敏感信息。
5. 通知责任人和受影响协作方。
6. 完成止血、修复和验证。
7. 恢复灰度并持续观察。
8. 形成复盘和预防措施。

复盘至少回答：

- 为什么需求、设计、开发、测试或审查阶段没有发现？
- 为什么监控没有提前发现？
- 为什么停止或回滚不够快？
- 哪条规则、测试、工具或责任需要调整？
- 改进项的负责人和期限是什么？

## 18. 防重复与无进展熔断

### 18.1 阻塞项台账

每个阻塞项至少记录：

```yaml
blocker_id: BLK-001
type: core-value | risk-control | repository-conflict | process-infrastructure
source_ref: ""
owner: ""
prerequisites: []
closure_evidence: []
attempt_count: 0
last_result: ""
remaining_gap: ""
```

### 18.2 熔断规则

- 一轮没有关闭任何 P0，下一轮必须重新规划，不得自动重发。
- 同一阻塞连续两次产生相同结果，禁止再次派发相同任务。
- 新任务必须说明变化的前提、方法、责任人或证据标准。
- 验证前提未就绪时，创建准备任务，不创建验证任务。
- 原始需求、派生风险、仓库冲突和流程基础设施必须分别标记，防止派生事项长期挤占核心用户价值。

## 19. 文件与证据分层

### 19.1 人工核心文件

- 已批准任务包。
- 实施计划。
- 代码差异和测试。
- 代码审查结论。
- 发布计划和回滚方案。
- 事故报告和复盘。

### 19.2 结构化人工证据

- 风险与未决项。
- 专项责任人意见。
- Gate 决策。
- 例外、豁免和延期记录。

### 19.3 机器维护文件

- 状态投影。
- 提交清单与哈希。
- 来源和追踪索引。
- CI 日志与检查结果索引。
- 看板渲染结果。

机器文件必须保留审计价值，但默认不进入人工主审视图。

### 19.4 建议目录模型

```text
sop/stages/04-development/
  dispatch/                 # 已批准开发任务包
  acceptances/              # 成员对精确任务版本的接受回执
  submissions/              # 成员完成报告与精确提交引用
  reviews/                  # PR、技术和专项审查结论
  integration/              # 合并、祖先校验和集成测试证据
  stage-state.yaml          # 机器状态投影

sop/stages/05-release/
  gate/                     # G4 人工审核包与决策
  rollout/                  # 灰度批次、指标和停止记录
  incidents/                # 事故与数据修复记录
  baseline/                 # 发布与 G5 关闭基线
```

目录名称可以适配现有仓库，但“人工核心文件、结构化决策、机器投影”三层不得混在同一份文档中反复人工确认。

## 20. 单任务完成定义（Definition of Done）

一项开发任务只有同时满足以下条件才可关闭：

- 已批准的验收标准全部满足。
- 代码修改未超出允许范围。
- 必需测试、lint、类型检查和构建通过。
- P0/P1 审查问题已经处理。
- 高风险专项审查已完成。
- 精确提交已集成到目标主干。
- 文档和迁移说明已更新。
- 风险、未运行项和遗留问题已如实记录。

单任务 `closed` 不代表项目可以发布；只有所有发布范围内任务完成、集成验证通过并获得 G4 批准后，才允许进入生产灰度。

## 22. 维护规则

- 本文档由团队技术负责人和项目责任人共同维护。
- 修改开发准入、人工 Gate、高风险权限或发布规则时必须经过人工确认。
